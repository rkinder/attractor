# Develop Design Document

## Context
You have completed the clarification interview and gathered sufficient requirements. Now you need to create a comprehensive design document using the EARS (Easy Approach to Requirements Syntax) methodology.

## Input

### Original Request
```
$user_request
```

### Generalized Problem
```
$generalized_problem
```

### Interview Summary
```
$interview_summary
```

## Your Task

Create a Design Document (`design.md`) following the EARS methodology.

## EARS Patterns Reference

| Pattern | Format | When to Use |
|---------|--------|-------------|
| **Ubiquitous** | The `<system>` shall `<action>` | Always applies, regardless of state/events |
| **Event-driven** | WHEN `<trigger>`, the `<system>` shall `<action>` | Triggered by an event or condition |
| **State-driven** | WHILE `<state>`, the `<system>` shall `<action>` | Only applies in a specific state |
| **Unwanted Behavior** | IF `<condition>`, THEN the `<system>` shall `<action>` | Error handling or failure response |
| **Optional Feature** | WHERE `<feature>` is included, the `<system>` shall `<action>` | Conditionally included features |

## Required Sections

### 1. Overview
What is being built and why? (2-3 paragraphs)

### 2. Architecture
- Components and their responsibilities
- Data flow
- Integration points
- Module boundaries

### 3. Functional Requirements
MUST use EARS patterns exclusively. For each requirement:

```
### FR-XXX: [Requirement Title]
**Type**: [Ubiquitous/Event-driven/State-driven/Unwanted/Optional]
**Statement**: [EARS-formatted requirement]
**Rationale**: [Why this requirement exists]
```

### 4. Non-Functional Requirements
- Performance
- Security
- Scalability
- Usability
- Reliability

### 5. Dependencies
- External systems
- Libraries/frameworks
- APIs
- Infrastructure

### 6. Open Questions
- Risks
- Unknowns
- Decisions needed

## Quality Checks

- [ ] Every functional requirement uses EARS pattern
- [ ] Architecture is clearly described
- [ ] Dependencies are identified
- [ ] Open questions are documented
- [ ] Requirements are testable (implicit in EARS structure)

## Output

Save to: `specs/[project-name]/design.md`

Format as markdown following the template in `.opencode/skills/ears-format/SKILL.md`
